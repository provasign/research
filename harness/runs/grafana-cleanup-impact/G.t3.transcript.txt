Based on my investigation, the `LifecycleManager` interface and its `Cleanup` method are confined entirely to the `pkg/registry/apps/annotation` package (no external references). I traced the interface, all three implementations, the caller chain, and the gRPC test mock.

Changing `Cleanup(ctx)` → `Cleanup(ctx, cutoff)` forces:
- The interface declaration (`storage.go`)
- All 3 implementations: `PostgreSQLStore`, `sqlAdapter`, `storeGRPC`
- The caller `runCleanup` (calls `lifecycleMgr.Cleanup`) and its caller `startCleanup` (which owns `retentionTTL` and must supply the cutoff)
- The test gRPC server mock `mockGRPCServer.Cleanup`, which calls `s.lifecycle.Cleanup`

`register.go` already passes `cfg.RetentionTTL` into `startCleanup`, so it needs no change. The generated proto methods keep their `*CleanupRequest` signatures (a cutoff field would be added to the message, not the function signature), so they are data/codegen changes, not function-signature sites.
