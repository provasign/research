I've traced the mechanism. The bug is an XORM session-state leak: `Statement.Table()` sets `AltTableName`, and `Statement.TableName()` returns `AltTableName` if non-empty — but `setRefBean()` (called by `Update`) only sets `tableName`, never clearing `AltTableName`. So when the alerting transaction reuses a session whose statement previously had `.Table("user")` set, an alert-rule `Update` that doesn't explicitly set its own table will emit SQL targeting `user` instead of `alert_rule`.

The only alert-rule `Update` calls (the ones lacking an explicit `.Table(...)`, unlike e.g. `admin_configuration.go:95`) are both in `alert_rule.go`:
- line 608 in `UpdateAlertRules`
- line 710 in `preventIntermediateUniqueConstraintViolations`

Both run on the shared transaction session and so both must explicitly specify the `alert_rule` table.
