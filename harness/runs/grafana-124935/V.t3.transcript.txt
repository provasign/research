I've traced the bug mechanism. In Grafana's xorm fork, `Session.Update` builds its SQL from `statement.TableName()` (session_update.go:215), which returns `AltTableName` if set (statement.go:523-528). `setRefBean` (called during `Update`) only resets `statement.tableName`, **not** `AltTableName` — so a leaked `AltTableName` (e.g. `"user"` from a prior `.Table("user")` query on a reused session) wins, producing `UPDATE/SELECT ... <alert_rule columns> ... user`. The `alertRule.TableName()` method does not protect against this; only an explicit `.Table(...)` on the call sets `AltTableName` correctly.

The runtime alert-rule update calls that lack an explicit table are both in `alert_rule.go` (lines 608 and 710). Other `.Update` calls in the store (`image.go`, `admin_configuration.go`) either target other tables or already pass `.Table(...)`. The migration site (`rule_group_index_migration.go:75`) shares the pattern but is a separate one-time code path, not the reported runtime failure.
